# NanoClaw 企业级落地方案 — Docker 单机架构

单台服务器 + Docker + Docker Compose，部署与运维尽量简单，同时保留扩展性，便于日后迁移到 K8s。

---

## 一、系统架构框架

采用"极简核心+可插拔扩展"架构，核心层保持精简，仅负责消息路由、容器管理、持久化存储，扩展层以模块化方式提供企业级功能。

```
┌─────────────────────────────────────────────────────────────┐
│                     企业应用层                                │
│  ┌────────────┐  ┌──────────────┐  ┌──────────────┐         │
│  │ 1对1助理    │  │ 审批流程助理   │  │ 项目小组助理   │         │
│  └────────────┘  └──────────────┘  └──────────────┘         │
└─────────────────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────────────────┐
│                     扩展层 (插件化)                           │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐        │
│  │ 技能系统  │ │权限管理    │ │记忆管理     │ Token监控 │        │
│  └──────────┘ └──────────┘ └──────────┘ └──────────┘        │
└─────────────────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────────────────┐
│                     核心层 (极简)                             │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐                   │
│  │消息路由   │  │容器管理    │  │持久化存储│                    │
│  └──────────┘  └──────────┘  └──────────┘                   │
└─────────────────────────────────────────────────────────────┘
```


| 层次      | 框架项         | NanoClaw 对应                                                                                                                    |
| ------- | ----------- | ------------------------------------------------------------------------------------------------------------------------------ |
| **核心层** | 消息路由        | Host 单进程：渠道回调 → 写入 SQLite → 轮询按 `chat_jid` 分组 → 分发到对应群组容器（[SPEC](../SPEC.md) Message Flow）。                                    |
|         | 容器管理        | `group-queue.ts` + `container-runner.ts`：按群组创建/复用 Docker 容器，空闲超时销毁，`MAX_CONCURRENT_CONTAINERS` 限并发。                            |
|         | 持久化存储       | 单 SQLite（`store/messages.db`）+ 按群组目录（`groups/{channel}_{name}/`、`data/sessions/{group}/`）；宿主机挂载卷，每群组独立子目录。                     |
| **扩展层** | 技能（Skill）系统 | `container/skills/` 下各技能（如 approval-system）；启动容器时同步到该群 `data/sessions/{group}/.claude/skills/`，Agent 在容器内按 SKILL.md/config 触发。 |
|         | 权限管理        | 无内置 RBAC；在 Skill 内调用企业 IAM 或配置文件做角色/属性校验；IPC 权限见 [IPC_PERMISSIONS_ZH.md](../IPC_PERMISSIONS_ZH.md)。                            |
|         | 记忆管理        | SQLite 存消息与路由状态；短期/长期记忆：群组 `CLAUDE.md` + `data/sessions/{group}/.claude/` 下 JSONL 会话与 projects。                                |
|         | 监控与告警       | 日志 stdout 由 Docker 收集；可选在网关或凭证代理侧统计 Token；简单脚本监控容器数量与进程健康。                                                                     |


## 二、部署架构（Docker Compose）

- **单台服务器**：安装 Docker、Docker Compose；无需 K8s。
- **Compose 编排建议**：
  - **NanoClaw 核心服务**（1 个容器）：运行 Host 进程（Node），挂载 Docker 套接字以按群组 `docker run` Agent 容器；挂载宿主机数据目录（见**四、存储设计**）。
  - **国产模型网关**（可选 1 个容器）：如 DashScope 网关，端口 8005；核心通过 `host.docker.internal` 或服务名访问。
  - **数据库**：沿用 **SQLite** 简化部署；`store/messages.db` 放在宿主机持久目录即可。
  - **Redis**：可选，用于后续会话缓存或限流；当前 NanoClaw 未内置，可预留。
- **Agent 容器**：由核心在运行时**动态创建**（非 Compose 定义），每群组一个，空闲超时后自动销毁；镜像使用 `container/build.sh` 构建的。

## 三、网络设计

- 使用 **Docker 自定义网络**（如 `nanoclaw_net`），核心容器与网关容器加入该网络，彼此通过服务名通信。
- Agent 容器由核心通过 `docker run --network nanoclaw_net`（或 host 网络，视凭证代理访问方式而定）启动；仅需能访问 Host 的凭证代理端口（如 3001）与网关端口（如 8005）。
- **对外暴露**：仅核心服务所需端口（若飞书等回调需经内网穿透或反向代理，则暴露该代理端口）；不对外暴露 Agent 容器端口。

## 四、存储设计（宿主机目录）

建议将 NanoClaw 的持久化根放在宿主机统一目录下，便于备份与权限管理：

```
/data/nanoclaw/                    # 宿主机持久化根（可挂载到核心容器为项目根）
├── store/                        # SQLite 与认证等
│   ├── messages.db               # 单库，按 chat_jid/group_folder 区分群组
│   └── auth/                     # 渠道认证（飞书不需要）
├── data/                         # 会话与 IPC
│   ├── env/env                   # 供 Agent 挂载的环境变量（从 .env 同步）
│   ├── sessions/                 # 按群组隔离
│   │   └── {group_folder}/
│   │       └── .claude/          # 会话、JSONL、skills 副本
│   └── ipc/
│       └── {group_folder}/       # input/, messages/, tasks/
├── groups/                       # 按群组隔离的记忆与日志
│   ├── CLAUDE.md                 # 全局记忆
│   └── {channel}_{name}/
│       ├── CLAUDE.md
│       └── logs/
├── logs/                         # Host 进程日志
│   ├── nanoclaw.log
│   └── nanoclaw.error.log
└── .env                          # 敏感配置（不提交），由环境或 Secret 注入
```

每个群组、1v1 对应独立 `groups/{channel}_{name}/` 与 `data/sessions/{group_folder}/`，实现租户/业务组级隔离。

## 五、安全框架

- **容器以非 root 运行**：NanoClaw Agent 镜像已以 `node` 用户运行（[SECURITY.md](../SECURITY.md)）。
- **Agent 容器资源与能力**：通过 `docker run` 可加 `--memory`、`--cpus` 限制；不授予多余 Linux Capabilities。
- **网络策略**：Agent 容器仅需访问 Host 凭证代理与模型网关；若技能需调外网（如 OA API），可单独放行或经 Host 代理转发。
- **敏感信息**：API Key、飞书 App Secret、OA 密钥等不写进代码库；通过环境变量或 Docker Secret 传入，核心写入 `data/env/env` 供 Agent 挂载（仅必要变量）；凭证代理内真实 Key 不落容器。

## 六、技能开发框架

- **技能形态**：NanoClaw 技能为**目录 + 描述文件**，运行在**同一 Agent 容器内**（非独立技能容器）。每群组一个 Agent 容器，容器内已挂载该群 `data/sessions/{group}/.claude/skills/`（从 Host 的 `container/skills/` 同步）。
- **技能描述**：`SKILL.md` 说明能力与触发方式；`config.json` 定义触发词、参数 schema、权限要求等（参见 `container/skills/approval-system`）。
- **与核心通信**：技能通过容器内 MCP（如 `mcp__nanoclaw__send_message`）或 IPC 与 Host 通信；不要求技能单独起 HTTP/gRPC 服务，除非企业自建扩展。

## 七、监控与日志

- **日志**：核心与 Agent 容器将日志输出到 stdout/stderr，由 Docker 收集；宿主机可将容器日志目录挂载到 `/data/nanoclaw/logs/`，并用 **logrotate** 做轮转。
- **简单监控**：脚本定期检查（1）核心容器是否在跑；（2）当前 Agent 容器数量与排队情况；（3）磁盘与 SQLite 大小；异常时告警（邮件/钉钉/飞书等）。

## 八、备份与恢复

- **定期备份**：对 `/data/nanoclaw/`（含 `store/`、`data/`、`groups/`、`logs/`）做定时备份；SQLite 建议在低峰期拷贝或 `sqlite3 .backup`。
- **备份脚本**：压缩后上传到对象存储或另一台服务器；保留最近 N 天版本。
- **恢复**：停止服务，解压备份到 `/data/nanoclaw/`，检查权限后重启 Compose；必要时执行 DB 迁移脚本（若有 schema 升级）。

## 九、升级与维护

- **升级**：拉取新镜像，更新 Compose 中镜像 tag，重启核心容器；Agent 容器会随下次消息自动使用新镜像（核心 `docker run` 时指定同一镜像名）。
- **数据库迁移**：若有 schema 变更，在部署流程中增加迁移步骤（如执行 `npm run migrate` 或脚本）。
- **回滚**：保留上一版本镜像；出问题时将 Compose 改回旧 tag 并重启。

## 十、成本控制

- **容器资源**：通过 `docker run` 的 `--memory`、`--cpus` 限制单 Agent 容器资源；`MAX_CONCURRENT_CONTAINERS` 限制并发容器数。
- **Token 配额**：在网关或凭证代理侧统计 Token，设置每日/每群组配额与告警；超限可拒绝或降级。
- **按需销毁**：Agent 容器在 `IDLE_TIMEOUT` 后自动退出，空闲时不占资源。

## 十一、实施步骤

1. 准备一台服务器，安装 Docker、Docker Compose。
2. 创建宿主机目录 `/data/nanoclaw/` 及子目录（store、data、groups、logs），设置权限。
3. 编写 Docker Compose：核心服务（挂载 Docker 套接字与 `/data/nanoclaw`）、可选网关；配置环境变量文件（不提交）。
4. 构建 Agent 镜像（`./container/build.sh`），核心启动时使用该镜像。
5. 启动核心服务；配置飞书等渠道与国产模型网关；执行 `cp .env data/env/env` 等同步。
6. 部署第一个技能（如审批）：将 `container/skills/approval-system` 放入项目，重启或等新容器拉起后即可使用。
7. 配置群组注册与权限（主群 + 测试群）；在 Skill 内按需对接 IAM。
8. 测试：飞书群内 @助手、审批流程走通；检查日志与审计。
9. 逐步上线其他技能与 1v1/项目群组；按需加监控与备份脚本。

## 十二、注意事项

- **备份必做**：SQLite 与目录需定期备份，避免单点丢失。
- **技能镜像**：Agent 镜像已包含通用技能目录；新增技能只需在 Host 的 `container/skills/` 增加目录，下次起容器自动同步，无需重做镜像（除非改 Dockerfile 依赖）。
- **日志巡检**：定期查看核心与容器日志，便于及早发现问题与错误。
