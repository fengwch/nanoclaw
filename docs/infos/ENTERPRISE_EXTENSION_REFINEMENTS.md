# 企业需求扩展分析 — 修正与落地说明

本文档对《NanoClaw 满足产品需求的分析》中需修正或需明确的部分做**统一表述**，便于落地到产品文档或设计说明时直接采用。仅包含已与当前 NanoClaw 实现对齐、且可执行的修正。

---

## 1. 记忆与权限相关「现状」表述

### 1.1 原表述（需修正）

- 「每个群组有**独立的数据库**」
- 「每个群组拥有独立的……对话历史存储在 SQLite 中」易被理解为「每群一个 SQLite 文件」

### 1.2 修正后表述（落地采用）

**记忆与存储现状：**

- NanoClaw 使用**单一 SQLite 库**（如 `store/messages.db`），通过 `chat_jid`、`group_folder` 等字段区分群组，实现**按群组/会话的行级隔离**，**并非**每个群组一个独立数据库文件。
- 每个群组拥有独立的**目录与文件**：`groups/{channel}_{name}/`（含 CLAUDE.md、日志、Agent 生成文件）、`data/sessions/{group_folder}/`（会话数据）；Docker 部署时上述路径挂载自 host，数据落 host。

在撰写「记忆机制」「权限控制」的现状时，请采用上述「单库 + 按群目录/会话隔离」的表述，避免基于「每群一库」做后续设计假设。

---

## 2. 权限控制配置的优先级

### 2.1 原表述（易导致先改核心）

- 「扩展群组模型：修改 `src/db.ts`，增加用户、角色、权限的表结构」

### 2.2 修正后表述（落地采用）

**权限控制配置建议顺序：**

1. **优先**：在 **Skill 内集成企业 IAM**（如 LDAP/AD 或企业统一身份 API），在每次工具调用前校验用户身份与权限；不修改 NanoClaw 核心表结构即可满足「用户级/角色级」控制与审计。
2. **可选/后续**：若确需在 NanoClaw 内持久化「用户–角色–权限」映射（如离线校验、缓存角色），再考虑扩展 `src/db.ts` 增加相应表结构；此时需与现有 `registered_groups`、会话表等兼容，并明确与 IAM 的同步策略。

文档中建议将「扩展 db 表」标为**可选或第二阶段**，并明确写清「优先在 Skill 中集成 IAM 与权限校验」。

## 3. 对话意图识别与 NLU 集成

### 3.1 原表述（易产生架构误解）

- 「将解析后的**结构化意图传递给 NanoClaw**」

### 3.2 修正后表述（落地采用）

**集成外部 NLU 的两种可行方式：**

- **方式 A（推荐，无需改核心）**：将 NLU 封装为 **Skill 中的工具**。由模型在需要时调用该工具，NLU 返回的结构化意图/槽位作为**工具结果**供模型使用，模型再据此决定调用其他工具或生成回复。输入仍为自然语言消息流，不改变 Host/Runner 的输入协议。
- **方式 B（需扩展）**：若必须在**请求前**就把结构化意图注入到 Agent（例如在构造发给模型的 first message 前就写入意图），则需要在 Host 或 container-runner 层增加**意图注入接口或扩展输入协议**，当前 NanoClaw 无此官方能力，需标注为「待设计/未验证」。

撰写时建议以「方式 A」为主，若涉及「方式 B」则单独注明「需扩展 Host/Runner 接口」。

## 4. Token 消耗监控的实现位置

### 4.1 原表述（过于具体且可能不准确）

- 「在 `src/container-runner.ts` 中……计算 Token 数量」

### 4.2 修正后表述（落地采用）

**Token 计量实现建议：**

- 在**模型 API 的调用链**上注入计量逻辑，具体位置需按实际代码确定，可能包括：
  - **Host 侧**：发起模型请求的 credential proxy 或调用处（如存在统一请求封装）；
  - **容器/Runner 侧**：若请求在 `container-runner` 或相关模块内发出，则在该路径上统计请求/响应的 Token 并上报；
  - **网关侧**：若使用自建网关（如 DashScope 网关），可在网关内解析请求与响应并统计 Token，再上报监控系统（如 Prometheus），无需改 NanoClaw 主仓代码。
- 文档中建议写为：「在模型 API 调用链上注入 Token 计量（Host 代理、container-runner 或网关侧择一或组合），并以实际代码调用路径为准；若使用自建网关，优先在网关侧实现以降低对主仓侵入。」
